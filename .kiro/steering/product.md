# Product Overview

LoomiDBX 是桌面端数据库模拟/合成数据生成工具，帮助用户连接目标数据库、扫描 Schema、配置字段生成规则、组织 Project，并将符合关系与约束的合成数据写回目标数据库。

本文件记录产品层面的长期约束和判断标准。它不是功能清单；新增功能只要符合这些原则，通常不需要更新本文件。

## Target Users

- **市场与售前人员**：快速准备演示数据，让 Demo 更接近真实业务场景。
- **数据库开发与测试人员**：构造可重复的测试数据，支持功能测试、性能测试和压测。
- **软件开发人员**：为本地开发、联调和测试环境填充数据，减少手工造数成本。

## Core Capabilities

### 数据库连接与 Schema 理解

产品需要支持用户管理多个数据库连接，并按用户触发扫描数据库结构。Schema 表达应覆盖数据库、Schema、表、字段、主键、外键、唯一性、非空、默认值等影响生成和写入的关键信息。

### 字段生成规则配置

字段规则描述“该字段最终应该生成什么样的数据”，而不是暴露底层实现细节。规则配置归属 Schema 层，重点表达字段类型、约束、生成器选择、参数和预览结果。

### Project 组织生成任务

Project 用于组织一次或一类可复用的生成任务。表级行数、清空策略、执行选项、表间数量关系和执行顺序归属 Project 层，不应混入字段规则模型。

### 关系与约束感知生成

生成数据必须优先满足数据库关系与约束，再追求展示效果。外键、唯一性、非空、默认值、枚举、范围等约束都应影响生成策略和执行预检。

### 执行、写回与历史

产品主路径是生成数据并写回用户目标数据库。执行过程应提供预检、进度、错误、结果和历史记录，便于用户复盘、重试和复用配置。

## Product Boundaries

## In Scope for First Release

- 数据库连接管理和连接验证。
- Schema 扫描、缓存和结构变更影响提示。
- 字段级生成规则配置和预览。
- Project 管理、表选择、行数配置和执行选项。
- 按依赖关系生成数据并写回目标数据库。
- 执行状态、执行结果和历史记录。
- 主要桌面 UI 工作流：登录、首页、项目、Schema 管理、设置、执行进度。

## Out of Scope for First Release

- AI 生成能力不进入首发主线；如后续纳入，应作为独立扩展能力或新阶段处理。
- 不在首期完整实现所有数据库方言；优先用 MySQL 和 PostgreSQL 验证抽象。
- 不做跨数据库迁移工具。
- 不追求覆盖所有高级数据库类型和特性，例如复杂数组、空间类型、用户自定义类型、分区表、物化视图等。
- 不把 Wails binding 当业务逻辑层。

## Value Proposition

LoomiDBX 的核心价值不是“随机填充数据”，而是生成更接近真实业务、可重复、可配置，并尽量满足数据库关系与约束的数据。

判断某个功能是否符合产品方向时，优先看它是否帮助用户完成以下闭环：

1. 理解目标数据库结构。
2. 用低认知成本配置字段和表级生成规则。
3. 在约束安全的前提下生成数据。
4. 将数据写回目标数据库。
5. 保存配置和结果以便复用。

## Product Rules

### 字段规则与执行规则分离

字段规则属于 Schema 层，表的生成数量、清空策略和执行选项属于 Project 层。不要为了 UI 方便把两者合并成一个不清晰的配置对象。

### 约束优先于展示效果

当格式化、随机性、唯一性、非空、默认值或外键关系冲突时，优先保证数据库约束成立。无法安全生成时应给出明确错误或预检提示，而不是静默降级。

### 外键字段优先遵从关联关系

外键字段应优先从被依赖表的已生成或已有数据中取值。不要把外键字段当作普通随机字段处理。

### 不可生成字段必须有依据

字段只有在允许为空、存在默认值、由数据库自动生成，或被明确排除且不会破坏约束时，才允许不参与生成。

### 依赖表先生成

存在外键依赖时，被依赖表应先于依赖表生成。循环依赖、缺失关系和无法排序的场景应在预检阶段暴露。

### 失败要可追踪

写入失败时应停止当前执行或当前批次，保留失败原因、影响范围和可供用户判断的执行结果。不要吞掉数据库错误。

### 结构变更必须提示影响范围

Schema 重新扫描后，应识别受影响的字段、规则和 Project 配置。用户需要知道哪些配置仍可复用，哪些需要调整。

## Privacy Boundary

LoomiDBX 是本地桌面应用。以下内容默认不得上传到远端账号服务：

- 数据库连接信息。
- Schema、表、字段和约束信息。
- 字段生成规则和 Project 配置。
- 生成数据内容。
- 用户 SQL。

远端账号服务只应处理登录、授权、会话、产品使用权和必要的非敏感遥测。任何新增远端能力都必须先明确数据边界。

## Spec and Phase Discipline

开发按 Phase 控制阅读范围和实现顺序，按 Phase 内小粒度 spec 执行。不要把 Phase 本身当成大 spec，也不要跨阶段提前实现未来能力。

当前首批实现聚焦 Phase 1：工程骨架、配置、本地存储、数据库方言接口和测试工具链。后续领域模型、执行引擎、生成器、API、UI 和验收按阶段推进。
