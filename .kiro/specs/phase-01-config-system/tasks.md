# Implementation Plan

- [ ] 1. 建立配置模型、默认值和错误契约
- [x] 1.1 定义应用配置模型和可持久化用户配置
  - 覆盖外观、路径、开发模式、未来账号/LLM 入口、隐私状态等应用级设置。
  - 明确普通配置、业务数据和敏感凭据的字段边界，避免把密钥类输入建模为普通明文字段。
  - 完成后，无配置文件场景可以构造出完整默认配置对象。
  - _Requirements: 1.1, 1.2, 1.3, 1.4_
- [x] 1.2 定义配置错误和设置视图契约
  - 提供字段路径、错误码、严重级别和用户可读消息，支持加载、校验和保存错误复用。
  - 提供后端 facade 和前端 API client 可使用的设置视图与更新输入。
  - 完成后，Go 与 TypeScript 侧都有不含 `any` 的精确设置 DTO。
  - _Requirements: 4.2, 5.1, 6.2, 6.3_

- [ ] 2. 实现路径发现、开发覆盖和文件存储基础
- [x] 2.1 实现配置文件路径和数据目录解析
  - 支持桌面默认路径、开发目录覆盖和测试隔离目录。
  - 对无效、非绝对、不可创建或不可写路径返回结构化 issue。
  - 完成后，默认、开发覆盖和测试模式都能返回确定性路径。
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 3.4_
- [x] 2.2 实现环境覆盖解析
  - 按固定名称读取当前 spec 支持的开发/测试覆盖值。
  - 对非法枚举、非法路径和不支持的敏感输入返回错误，不静默忽略。
  - 完成后，环境覆盖可以覆盖文件配置，但不会被标记为需要写回。
  - _Requirements: 2.3, 3.1, 3.3, 3.4_
- [x] 2.3 实现普通配置文件读写
  - 文件缺失时返回 missing 状态，由加载流程使用默认配置继续。
  - 保存时创建父目录，并采用临时文件替换方式减少半写入风险。
  - 完成后，写入失败不会留下无法解析的目标配置文件。
  - _Requirements: 3.2, 5.2, 5.3, 5.4_

- [ ] 3. 实现配置加载、校验和保存服务
- [x] 3.1 实现默认值、文件配置和环境覆盖的合成流程
  - 固定加载顺序为默认值、配置文件、环境覆盖。
  - 返回配置来源状态，让调用方可区分首次启动、文件加载和覆盖生效。
  - 完成后，缺失配置文件不会导致启动失败，最终配置仍完整可用。
  - _Requirements: 1.1, 3.1, 3.2, 5.1_
- [x] 3.2 (P) 实现配置校验规则
  - 校验语言、主题、运行模式、路径、未来入口状态和敏感明文禁止规则。
  - 加载和保存都复用同一套校验逻辑。
  - 完成后，无效配置会返回字段级 issue，且不会作为有效配置继续使用。
  - _Requirements: 1.3, 1.4, 2.4, 3.3, 4.1, 4.2, 4.4_
  - _Boundary: ConfigValidator_
- [x] 3.3 实现配置服务的读取与更新行为
  - 提供后端不依赖 Wails UI 的读取入口。
  - 保存有效更新时只持久化用户可变部分，保留未修改字段的默认值或现有值。
  - 完成后，保存后再次加载会得到与用户可变项一致的配置视图。
  - _Requirements: 5.1, 5.2, 5.4, 6.1_
  - _Depends: 2.1, 2.2, 2.3, 3.1, 3.2_

- [ ] 4. 接入 Wails facade 与前端设置契约
- [x] 4.1 接入后端配置 facade
  - 在应用组合根中初始化配置服务，并暴露读取设置和更新设置的薄入口。
  - facade 只做参数转发和错误转换，不实现配置合成、路径解析或校验规则。
  - 完成后，Go 侧可通过 facade 获取设置视图或字段级配置错误。
  - _Requirements: 6.1, 6.2, 6.3, 6.4_
  - _Depends: 3.3_
- [x] 4.2 (P) 建立前端设置类型和 API client
  - 定义设置视图、更新输入、配置 issue 等 TypeScript 类型。
  - 封装 generated settings binding，并转换为现有 `ApiResult` 风格。
  - 完成后，前端无需设置页面即可通过 client mock 验证成功和错误分支。
  - _Requirements: 6.2, 6.3, 6.4_
  - _Boundary: SettingsClient_
  - _Depends: 1.2_

- [ ] 5. 验证隐私边界、保存一致性和相邻 spec 集成点
- [x] 5.1 编写配置系统单元测试
  - 覆盖默认值、路径解析、环境覆盖、加载顺序、校验错误和保存后再次加载。
  - 验证敏感字段不会写入普通配置文件，错误信息不包含敏感原值。
  - 完成后，配置包测试不依赖网络、目标数据库、SQLite schema 或真实用户目录。
  - _Requirements: 1.1, 1.4, 2.1, 2.2, 2.4, 3.1, 3.2, 3.3, 4.1, 4.2, 4.4, 5.3, 5.4_
- [x] 5.2 编写 facade 与前端契约测试
  - 验证 facade 通过配置服务返回设置视图和校验错误。
  - 验证前端 API client 不直接把页面绑定到 generated binding 细节。
  - 完成后，Go facade 和 TypeScript client 的契约可独立验证，且不要求完整设置页存在。
  - _Requirements: 5.1, 6.1, 6.2, 6.3, 6.4_
  - _Depends: 4.1, 4.2_
- [x] 5.3 验证本地存储策略可复用的数据目录输出
  - 使用测试替身读取配置系统输出的数据目录，不创建 SQLite schema 或迁移。
  - 确认配置系统文档和错误消息都把 SQLite 业务存储标注为相邻 spec 范围。
  - 完成后，后续本地存储 spec 可以依赖配置数据目录而不反向修改配置加载规则。
  - _Requirements: 2.2, 5.1, 6.1_
  - _Depends: 3.3_

- [ ] 6. 收尾配置模块文档和边界标注
- [x] 6.1 更新配置模块说明
  - 将 `internal/config` 从占位说明更新为配置模型、路径、加载、保存和校验的边界说明。
  - 标注完整设置页、SQLite schema、安全凭据存储和远端账号能力的归属。
  - 完成后，开发者能从模块说明判断新增配置相关代码应放在哪里，以及哪些内容不属于本 spec。
  - _Requirements: 1.3, 1.4, 4.3, 4.4, 6.4_
- [x] 6.2 运行基础验证并确认范围未外溢
  - 运行 Go 测试和前端类型检查中与配置契约相关的验证。
  - 检查配置实现没有网络上传、目标数据库访问、SQLite schema 创建或完整设置页实现。
  - 完成后，验证结果证明配置系统满足当前 spec，且未吸收相邻 spec 的职责。
  - _Requirements: 4.3, 5.1, 5.2, 5.3, 5.4, 6.1, 6.4_
