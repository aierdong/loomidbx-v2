# LoomiDBX 用户故事地图

## 1. 文档目的

本文档用于将 LoomiDBX 的产品大纲进一步拆解为用户故事地图，重点回答“用户要做什么”，并按用户旅程组织功能任务与主流程。 
该文档的目标是为后续的 MVP 范围划分、版本排期和详细需求拆解提供基础。 

## 2. 用户角色

LoomiDBX 当前的目标用户主要分为三类：市场/售前人员、数据库开发/测试人员、软件开发人员。 

| 用户角色 | 典型场景 | 核心目标 | 关注重点 |
|---|---|---|---|
| 市场、售前人员 | 演示、方案展示、客户沟通 | 快速生成可展示的 Demo 数据 | 数据外观真实、准备时间短、演示可控   |
| 数据库开发、测试人员 | 测试验证、性能测试、压力压测 | 构造符合约束的测试数据 | 关系一致性、唯一性、可重复执行、批量生成   |
| 软件开发人员 | 开发环境填充、联调准备 | 快速补充开发所需数据 | 降低人工造数成本、便于复用配置   |

## 3. 用户目标

从产品定位看，LoomiDBX 的核心目标是生成“逼真、可复用、符合关系与约束”的模拟数据，并回填到目标数据库中。 

| 目标层级 | 用户期望 | 对产品的要求 |
|---|---|---|
| 结果目标 | 快速得到可用数据 | 支持生成并写入数据库   |
| 质量目标 | 数据看起来像真实业务数据 | 支持随机、分布、枚举、计算、外键关联等规则   |
| 正确性目标 | 不破坏数据库约束 | 保证主键、外键、唯一、非空等约束成立   |
| 效率目标 | 下次可以继续用 | 保存连接、字段规则和 Project 配置   |

## 4. 主流程

根据产品大纲，用户的主流程可以概括为从“连接数据库”到“生成并复用配置”的完整闭环。 

| 阶段 | 用户在做什么 | 主要产出 |
|---|---|---|
| 1. 连接数据库 | 创建连接、填写连接信息、验证可用性 | 可访问的数据库连接   |
| 2. 扫描结构 | 识别数据库、Schema、表、字段、关系和约束 | 可浏览的数据库结构   |
| 3. 配置规则 | 为字段定义生成方式并保存规则 | 字段生成配置   |
| 4. 组织项目 | 把表组合为 Project 并配置执行参数 | 可执行的生成任务   |
| 5. 执行生成 | 按依赖顺序生成数据并写回数据库 | 已写入的模拟数据   |
| 6. 查看结果 | 查看进度、结果、历史记录 | 任务结果记录   |
| 7. 复用调整 | 复用已有配置，处理结构变更 | 可再次运行的配置资产   |

## 6. 分阶段任务说明

### 6.1 准备连接

这一阶段的目标是让用户顺利进入系统，并建立可用的数据库访问入口。 

| 任务 | 用户意图 | 成功标准 |
|---|---|---|
| 新建连接 | 接入一个数据库环境   | 连接信息被正确保存 |
| 编辑连接 | 更新地址或账号等信息   | 修改后可重新验证 |
| 删除连接 | 清理不再使用的环境   | 连接从列表中移除 |
| 验证连接 | 确认数据库是否可访问   | 验证结果明确反馈 |

### 6.2 理解结构

这一阶段的目标是帮助用户理解数据库结构，并为后续字段级规则配置提供依据。 

| 任务 | 用户意图 | 成功标准 |
|---|---|---|
| 浏览层级结构 | 快速定位数据库对象   | 能按数据库、Schema、表、字段逐层查看 |
| 查看字段属性 | 判断字段约束与可生成性   | 显示类型、主键、外键、非空、唯一性等信息 |
| 重新扫描结构 | 应对结构变化   | 新结构被识别出来 |
| 提示影响范围 | 判断哪些配置需要调整   | 用户能看到受影响字段与规则 |

### 6.3 定义规则

这一阶段的目标是把“字段最终生成什么样的数据”表达清楚，而不是让用户理解底层生成实现。 

| 任务 | 用户意图 | 成功标准 |
|---|---|---|
| 选择生成方式 | 决定字段的数据来源和形态   | 字段被绑定到一种生成方式 |
| 配置规则参数 | 控制数据分布、格式和范围   | 规则可保存、可复用 |
| 预览配置效果 | 先确认结果再执行   | 用户能看到字段生成预览 |
| 保存字段规则 | 形成可复用配置资产   | 规则可用于后续 Project |

### 6.4 组织任务

这一阶段的目标是把多个表组织成一次可执行的生成任务。 

| 任务 | 用户意图 | 成功标准 |
|---|---|---|
| 创建 Project | 建立一次任务容器   | Project 可被保存和编辑 |
| 选择表 | 确定本次参与生成的范围   | 表被加入 Project |
| 设置数量 | 控制写入规模   | 每张表的生成数清晰可见 |
| 设置清空策略 | 决定是否覆盖原数据   | 执行前选项明确 |
| 设置数量关系 | 保持关联表的数据比例   | 关联表数据量符合预期 |
| 预览任务 | 检查执行顺序和参数 | 用户可在执行前确认无误 |

### 6.5 执行生成

这一阶段的目标是按正确顺序写入数据，并在失败时保留结果状态。 

| 任务 | 用户意图 | 成功标准 |
|---|---|---|
| 启动任务 | 正式开始写入   | 任务进入执行状态 |
| 按依赖生成 | 确保外键关系成立   | 被依赖表先生成 |
| 查看进度 | 掌握执行过程   | 进度信息可见 |
| 中止失败批次 | 避免错误继续扩散   | 失败结果被保留 |

### 6.6 查看复用

这一阶段的目标是把执行结果变成可追踪、可复用的资产。 

| 任务 | 用户意图 | 成功标准 |
|---|---|---|
| 查看结果 | 确认是否成功写入   | 成功/失败状态清楚 |
| 查看历史 | 回看过去执行记录   | 记录包含任务名、Project、时间、数量等信息 |
| 复用配置 | 降低再次配置成本   | 已保存配置可再次调用 |
| 调整受影响项 | 处理结构变化后的配置偏差   | 受影响规则被重新修正 |

## 7. MVP 边界

MVP 聚焦在“连接数据库、扫描结构、配置字段规则、组织 Project、执行生成、查看历史”这条主链路上。 

| 优先级 | 范围 | 说明 |
|---|---|---|
| 必须 | 数据库连接管理   | 没有连接就无法进入后续流程。 |
| 必须 | Schema 扫描与字段浏览   | 是规则配置和任务组织的基础。 |
| 必须 | 字段规则配置   | 是产品核心价值。 |
| 必须 | Project 组织与执行   | 是从配置到落库的闭环。 |
| 必须 | 历史记录保存   | 支撑复用与追踪。 |
| 暂缓 | 数组类型输出   | 已明确不纳入当前范围。 |
| 暂缓 | AI 生成能力首发   | 已明确不纳入当前范围。 |
| 后续专题 | 异常处理、API 契约、执行引擎等   | 留待后续专题展开。 |
